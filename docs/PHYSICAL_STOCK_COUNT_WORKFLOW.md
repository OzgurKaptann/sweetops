# Physical Stock Count Workflow

A manager walks to the freezer, counts what is actually in it, and tells SweetOps
what they found. The system compares that against what it believed, shows the
difference, and corrects itself — keeping the count, the system's own figures, and
the correction as one linked, auditable event.

This document explains why that is a first-class workflow and not a manual
adjustment with a nicer form.

---

## 1. Why a physical count is not a manual adjustment

SweetOps already had a way to change on-hand stock: `MANUAL_ADJUSTMENT`, a signed
delta with a reason. A count could, technically, be typed in that way. This
feature exists because doing so destroys most of what a count is for.

A manual adjustment is a **one-off correction**. Someone knows a number is wrong,
and types the difference. It records the correction.

A physical count is an **observation**. Someone went and looked. What a manager
needs to be able to say afterwards is:

> On the 14th, Kadıköy counted the chocolate freezer. The system said 4.200 kg.
> The shelf held 3.850 kg. We were 0.350 kg short.

A bare `-0.350` in the ledger cannot say that:

* **It throws away the two numbers the difference came from.** Nobody can check
  the arithmetic afterwards, and nobody can tell a real count from a guess. The
  ledger says the stock was corrected; it does not say the shelf was ever looked at.
* **A shelf that was counted and found CORRECT leaves no trace at all.** Its delta
  is zero, and a zero-delta adjustment is rejected as a no-op. "We counted it and
  it was right" becomes indistinguishable from "nobody looked" — and for stock
  control that is exactly the wrong pair of facts to conflate.
* **It hides shrinkage.** To analytics, a counted discrepancy and a deliberate
  correction are the same row. An owner cannot see how much stock is quietly
  walking out of the door, because the ledger cannot tell them apart.
* **Nothing binds the correction to the act of counting.** There is no evidence
  behind the number.

So a count is a row in its own table, holding the three numbers **as a set**, and
the ledger movement — if there is one — points back at it.

**The count is the event. The movement is its stock effect.** The database refuses
to let the two disagree.

## 2. The workflow

```
count what is physically present
compare it against system on-hand quantity
show the difference
apply a controlled count adjustment
preserve the count and its reconciliation history
```

Concretely, one count:

1. The manager picks the ingredient and enters **what they physically counted**.
   Not a delta — the actual quantity on the shelf.
2. The server locks that store's stock row and reads **what it believes right now**:
   `system_on_hand_quantity` and `system_reserved_quantity`.
3. It computes `delta = counted − system_on_hand`. The client never supplies this.
4. If `counted < reserved`, it **refuses** (see §5).
5. It writes the count row, and — only if the delta is non-zero — one
   `STOCK_COUNT_ADJUSTMENT` movement carrying the count's id.
6. It sets `on_hand_quantity` **to** the counted figure. `reserved_quantity` is not
   touched.
7. It audits the whole thing as `INVENTORY_STOCK_COUNTED`.

All of it in one transaction. A deferred trigger re-checks at COMMIT that the count
and its movement agree. There is no draft, no pending, no half-applied count.

### Why the client does not compute the delta

The delta is measured against what the system believed **at the instant the count
was applied**, under a row lock. A client-computed delta would be measured against
whatever the manager's browser last rendered — and an order placed thirty seconds
ago has already made that stale. The request body has no field for the delta, and
unknown fields are rejected outright.

## 3. Data model

### `inventory_stock_counts`

| Column | Notes |
|---|---|
| `id` | |
| `store_id` | From the session. Never the request body. |
| `ingredient_id` | |
| `counted_quantity` | What was physically on the shelf. May be **0**. |
| `system_on_hand_quantity` | What the system believed, captured under the lock. |
| `system_reserved_quantity` | What was already promised to accepted orders. |
| `delta_quantity` | **GENERATED** `counted − system_on_hand`. Not writable. |
| `unit` | |
| `reason` | Mandatory. |
| `note` | Optional. |
| `status` | `APPLIED`, and only `APPLIED`. |
| `counted_by_user_id` | Must belong to `store_id`. |
| `idempotency_key_hash` | SHA-256. The raw key is never stored. |
| `request_hash` | SHA-256 of the request body. |
| `created_at`, `applied_at` | |

`delta_quantity` is generated by PostgreSQL rather than written by the application.
A count that claimed a delta its own two numbers do not support would be precisely
the lie this table exists to prevent, so it is not a row that can be stored.

### Status

There is **one** status, `APPLIED`, enforced by a CHECK constraint. The count row
and its movement post atomically, so there is no intermediate state to represent.

`DRAFT` / `SUBMITTED` / `APPROVED` are deliberately **not** in the domain. A mutable
status column with no state machine behind it is a lie that invites one to be
written later. When a real approval flow is built, the domain widens then — see §12.

### Immutability

`inventory_stock_counts` is append-only: UPDATE and DELETE are refused by a trigger,
exactly as for the movement ledger. A count that was got wrong is **not edited** —
today's manager does not get to rewrite what yesterday's manager says they saw on
the shelf. It is superseded by counting again, which is what one would physically
have to do anyway.

## 4. The movement type

`STOCK_COUNT_ADJUSTMENT` — the signed delta that makes system on-hand equal what
was counted.

```
system_on_hand = 10.000            system_on_hand = 10.000
counted        =  9.250            counted        = 11.500
delta          = -0.750            delta          = +1.500

movement_type          = STOCK_COUNT_ADJUSTMENT
quantity               = 0.750     quantity               = 1.500
quantity_delta_on_hand = -0.750    quantity_delta_on_hand = +1.500
quantity_delta_reserved = 0        quantity_delta_reserved = 0
```

A count corrects in **either** direction — the shelf may hold more than the system
thought — so the delta rule is `abs(quantity_delta_on_hand) = quantity`, and
`quantity_delta_reserved` is pinned to `0`.

It is its own type, not a `MANUAL_ADJUSTMENT`, for the same reason the transfer legs
are their own types: **analytics must be able to tell the two apart.** A count says
"we looked, and reality differs". An adjustment says "we know this figure is wrong".
Folding the first into the second loses the count's two system snapshots and makes
shrinkage invisible.

### Zero-delta policy — **the count is recorded, and writes no movement**

This is the chosen policy, and it is the reason the table exists.

When the shelf agrees with the system:

* the **count row is written** — it is the evidence that somebody looked;
* **no movement is written** — nothing physical happened, so nothing belongs in the
  physical ledger. A zero-delta movement is a ledger row that moves no stock, which
  is noise in the one record an auditor reads.

The API returns `movement_id: null`, and the UI reports
**"Sayım kaydedildi. Stok farkı oluşmadı."** — a success with its own message, not a
failure and not a replay.

Note that the schema alone almost enforces this. A `STOCK_COUNT_ADJUSTMENT` must
satisfy `abs(quantity_delta_on_hand) = quantity`, and the pre-existing
`ck_movement_quantity_positive` requires `quantity > 0`. A zero-delta movement would
need `quantity = 0` and is therefore **structurally unrepresentable**. The deferred
trigger enforces the other half: a non-zero count with no movement cannot commit
either.

## 5. The reserved-stock safety rule

```
counted_quantity must be >= system_reserved_quantity
```

Refused with `409 stock_count_below_reserved`:

> **Sayım sonucu, bekleyen siparişler için ayrılmış stoktan düşük olamaz.
> Önce ilgili siparişleri kontrol edin.**

If a manager counts 3 kg of chocolate while 5 kg is already promised to accepted
orders, the honest reading is **not** "the system was wrong by 2 kg". It is:

> **This shop has sold 2 kg of chocolate it does not physically have.**

That is not a stock correction. Writing on-hand down to 3 would break
`ck_stock_reserved_le_on_hand` anyway; suppressing that by silently releasing
reservations would break a promise to a customer who is sitting at a table waiting
for their waffle. Neither is something a stock form should do quietly.

It is an **operational incident**, and it needs a human: cancel or re-source the
affected orders, *then* count. The database says so
(`ck_stock_count_counted_ge_reserved`), the service says so, and the UI says so
before the manager even submits.

Counting **exactly** equal to reserved is allowed — the rule is `>=`, not `>`. Every
gram on the shelf is promised and none is missing, which is tight but not a
contradiction.

## 6. Stock effects

```
on_hand   := counted_quantity        (via the signed STOCK_COUNT_ADJUSTMENT)
reserved  :  UNCHANGED
available :  changes ONLY because on_hand did
```

`available_quantity` is a generated column (`on_hand − reserved`), so it is never
written directly. A count that lowers on-hand lowers available by the same amount.

**Reserved never moves.** A count observes what is physically on the shelf; it says
nothing about what has been promised. A manager counting the freezer has no business
silently un-promising a waffle to the customer at table four — and if the count
genuinely cannot accommodate the promises, that is §5, not a silent release.

| Case | on-hand | reserved | movement |
|---|---|---|---|
| counted < system | falls to counted | unchanged | one, negative |
| counted > system | rises to counted | unchanged | one, positive |
| counted = system | unchanged | unchanged | **none** |
| counted < reserved | **refused** | **refused** | none |
| counted = 0, reserved = 0 | falls to 0 | unchanged | one, negative |

Counting an **empty shelf is valid**, and is the count a manager most needs to be
able to report. The counted quantity may be zero; it may never be negative.

## 7. Idempotency

Every count requires an `Idempotency-Key` header.

* same store + same key + **same** payload → returns the original count,
  `idempotent_replay: true`, and **no further stock moves**;
* same store + same key + **different** payload → `409 idempotency_mismatch`.
  Replaying the original would tell a manager who re-counted and found 8 kg that
  their 9.25 kg count succeeded — and they would walk away believing the shelf was
  reconciled;
* **a different store may reuse the same key freely.** Two branch managers working
  from the same printed count sheet will legitimately send the same key. That is a
  coincidence, not a replay. Uniqueness is `(store_id, idempotency_key_hash)`.

The raw key is **never stored** — only a SHA-256 digest — and it is never echoed
back in a response. A replayable credential in a log or a response body is a
liability, not a control.

## 8. Audit

Action: **`INVENTORY_STOCK_COUNTED`**, entity type `inventory_stock_count`.

Payload:

```
stock_count_id, store_id, ingredient_id,
counted_quantity, system_on_hand_quantity, system_reserved_quantity,
delta_quantity, unit, actor_user_id, reason, status
```

**Never logged:** the raw idempotency key, the request hash, the CSRF token, the
session token.

## 9. Reconciliation

`scripts/reconcile_inventory.py` — read-only, store-scoped, and it **never writes**.

A count's correction is an **ordinary on-hand delta**. It is not special-cased, not
excluded, and not exempt from the arithmetic: check 1 (summary vs ledger) sums the
`STOCK_COUNT_ADJUSTMENT` alongside every other movement, and the store reconciles on
its own. A count saying the shelf holds 350 g less really does mean the shelf holds
350 g less.

What a per-store total **cannot** see is a count whose movement is missing or wrong.
That is check 5:

```
delta <> 0  ⟹  exactly ONE matching STOCK_COUNT_ADJUSTMENT
delta  = 0  ⟹  exactly ZERO movements
```

A count row claiming the shelf was corrected, with no movement behind it, leaves the
ledger and the summary in perfect agreement **with each other** — both simply still
hold the pre-count figure. Nothing in any total is wrong. The count sheet and the
ledger tell different stories and each looks internally consistent on its own. Only
comparing the count against its movement finds it.

The database refuses to create that (the deferred trigger checks it at COMMIT), so
the check exists to catch what got in some **other** way: a manual SQL edit, a
restore from an inconsistent backup, a future migration bug.

The report names the `stock_count_id`, so an operator can go and look at it.

**Zero-delta counts are never reported as drift.** The shelf agreed with the system;
that is the healthy case, and recording it is the whole point.

## 10. Analytics

A counted discrepancy is **its own correction type**. It must never be counted as:

| Not this | Because |
|---|---|
| **waste** | The branch did not throw the chocolate away. The waste report would accuse it of a loss it did not incur. |
| **purchase receipt** | Nothing was bought. The purchasing report would invent a supplier delivery. |
| **consumption** | Nobody ate it. Consumption velocity drives every reorder decision, and would be inflated by stock that was never sold. |
| **transfer** | It did not go to another branch. |
| **manual adjustment** | Nobody decided to change the number. Somebody *counted*, and that distinction is the entire feature. |

Consumption velocity in `decision_engine` filters on `movement_type = CONSUMPTION`,
so `STOCK_COUNT_ADJUSTMENT` is excluded by construction.

A count **does** change the shelf, so the **stock summary and stockout risk reflect
it immediately.** Being excluded from the flow metrics and being reflected in the
stock level are not in tension: one is about what *happened*, the other about what
*is*.

In the movement ledger the type renders as **"Sayım düzeltmesi"**. The raw enum is
never displayed.

## 11. API

All under the inventory router. State-changing routes require an authenticated staff
session, a trusted Origin, a valid CSRF token, `inventory:adjust`, and an
`Idempotency-Key`. Reads require `inventory:read` and a store-assigned session.

```
POST /inventory/stock-counts
GET  /inventory/stock-counts
GET  /inventory/stock-counts/{id}
```

### Request

```json
{
  "ingredient_id": 3,
  "counted_quantity": "3.850",
  "reason": "Haftalık dolap sayımı",
  "note": "2 numaralı dolap"
}
```

`extra="forbid"`. A body carrying `store_id`, `delta_quantity`,
`system_on_hand_quantity`, `actor_user_id`, `movement_type`, `idempotency_key_hash`,
`request_hash` or `status` is **rejected with a 422** — not silently ignored.
Silently ignoring a smuggled `store_id` would leave a client believing it had counted
another branch's freezer, and cheerfully told so.

The store comes from the session. There is deliberately no field to set it: the
**absence of the parameter is the security property**.

### Response

```json
{
  "stock_count_id": 12,
  "store_id": 1,
  "ingredient_id": 3,
  "counted_quantity": "3.850",
  "system_on_hand_quantity": "4.200",
  "system_reserved_quantity": "0.500",
  "delta_quantity": "-0.350",
  "movement_id": 91,
  "status": "APPLIED",
  "on_hand_quantity": "3.850",
  "reserved_quantity": "0.500",
  "available_quantity": "3.350",
  "created_at": "...",
  "applied_at": "...",
  "idempotent_replay": false
}
```

`movement_id` is **null** for a zero-delta count.

### Errors

| Code | Status | Meaning |
|---|---|---|
| `stock_count_below_reserved` | 409 | Counted below what accepted orders are promised. |
| `stock_not_configured` | 404 | This branch has no stock row for that ingredient. |
| `ingredient_not_found` | 404 | Unknown or inactive ingredient. |
| `idempotency_required` | 400 | No `Idempotency-Key`. |
| `idempotency_mismatch` | 409 | Same key, different payload. |
| `stock_count_not_found` | 404 | No such count **in this store**. |
| `no_store_assigned` | 403 | Staff session with no branch. |

`stock_count_not_found` is a 404 and not a 403 on purpose: another branch's count is
none of the caller's business, and a 403 would confirm it exists.

## 12. Permissions

No new permission was added.

| Role | Read counts | Apply a count |
|---|---|---|
| OWNER | ✅ | ✅ |
| MANAGER | ✅ | ✅ |
| KITCHEN | ✅ (`inventory:read`) | ❌ |
| CASHIER | ❌ | ❌ |

`inventory:adjust` is the physical-stock authority — the same one waste, manual
adjustment and transfer need, and for the same reason: this permanently changes what
a branch believes is on its shelves. A cook correcting the count is exactly the
unaccountable adjustment the inventory lifecycle exists to prevent, so KITCHEN may
look and may not touch. CASHIER has no inventory permission at all: money and stock
are separate authorities.

**There is no cross-store count.** `fk_stock_count_actor_store` binds the counter to
the store whose shelf they counted, so a Kadıköy manager counting Beşiktaş's freezer
is not a permission check that could be got wrong — it is a row PostgreSQL will not
store.

## 13. Database constraints

| Constraint | Makes unrepresentable |
|---|---|
| `ck_stock_count_counted_nonneg` | A shelf holding a negative amount. |
| `ck_stock_count_counted_ge_reserved` | A count below what is already promised. |
| `ck_stock_count_status_domain` | A fake workflow state (`DRAFT`, `APPROVED`, …). |
| `ck_stock_count_reason_present` | An unexplained correction to physical stock. |
| `fk_stock_count_actor_store` | Counting a branch you do not work at. |
| `fk_stock_count_stock_store` | Counting a shelf the branch does not have. |
| `uq_stock_count_store_idem` | A replay within one store (and *only* within one store). |
| `fk_movement_stock_count_leg` | A count's correction booked against another store or ingredient. |
| `ck_movement_stock_count_link` | A `STOCK_COUNT_ADJUSTMENT` with no count behind it — **and** a count id on a WASTE or receipt, which would let a write-off masquerade as a counted discrepancy. |
| `uq_movement_stock_count` | The same correction posted twice. |
| `ck_movement_delta_matches_type` | A movement whose quantity or direction disagrees with its own delta, or that moves reserved stock. |
| `delta_quantity` GENERATED | A count claiming a delta its own two numbers do not support. |
| deferred trigger | A non-zero count with **no** movement; a zero-delta count **with** one; a movement that does not match its count. |
| immutability trigger | Editing or deleting a count after the fact. |

The deferred constraint trigger is written to the same rules as the append-only and
transfer-pairing triggers beside it: `SECURITY INVOKER`, a pinned `search_path`,
schema-qualified references, no dynamic SQL, `EXECUTE` revoked from `PUBLIC`. There
is **no application-reachable bypass** — no GUC and no session variable turns it off,
because any role (including one reached through an injection path) could set one.

## 14. Owner UI

**Stok Yönetimi → "Sayım gir"** (dialog: **"Fiziksel sayım"**).

It sits directly beside the other stock operations, and ahead of "Manuel düzeltme",
on purpose: a count that is harder to reach than a manual adjustment is a count that
gets **typed in as** a manual adjustment.

Fields: **Malzeme**, **Sayım sonucu**, **Sebep**, **Not** (optional).

> **Bu işlem fiziksel stok miktarını sayım sonucuna göre düzeltir.
> Ayrılmış stok değişmez.**

The second sentence is the operational one. A manager who fears that counting the
freezer might cancel a waiting customer's waffle will not count the freezer.

When an ingredient is selected, the form shows the figures being reconciled against:

* **Sistemdeki fiziksel stok**
* **Ayrılmış stok**
* **Kullanılabilir stok**
* **Beklenen fark** — computed live as `counted − system on-hand`

The expected difference is a **preview only**. The server recomputes it from the row
it locks, and its answer is the one that counts.

| State | Copy |
|---|---|
| Success | **Sayım kaydı uygulandı.** |
| Zero delta | **Sayım kaydedildi. Stok farkı oluşmadı.** |
| Replay | **Bu sayım daha önce kaydedilmiş. Stok yeniden düzeltilmedi.** |
| Below reserved (client) | **Sayım sonucu ayrılmış stoktan düşük olamaz.** |
| Below reserved (server) | **Sayım sonucu ayrılmış stoktan düşük olamaz. Ayrılmış stok bekleyen siparişler için tutuluyor; önce ilgili siparişleri kontrol edin.** |
| Outcome unknown | **Sayım sonucunun kaydedilip kaydedilmediği doğrulanamadı. Aynı işlemi tekrar göndermeden önce stok hareketlerini kontrol edin.** |

The uncertain-network copy is deliberately **not** phrased as a failure. The
endpoints are idempotent, so an unchanged retry is safe — but a manager who reads
"başarısız" will re-enter the form by hand, which mints a **new** idempotency key and
genuinely doubles the correction. So: go and look at the ledger first.

The client-side below-reserved block is a **courtesy, not a control**. The service
enforces it, and its answer is the one that counts.

## 15. Operations

Per store, per count cycle:

1. **Count the shelf.** Freezer, container, dry store — whatever the cycle covers.
2. **Compare.** Open *Stok Yönetimi → Sayım gir*, pick the ingredient, and read
   *Sistemdeki fiziksel stok* against what is in your hand.
3. **Apply.** Enter the counted quantity and a reason. Check *Beklenen fark* before
   submitting — it is the correction that is about to be applied.
4. **If the count is below reserved,** the system will refuse. That is not a bug: the
   branch has accepted orders it cannot physically fill. Resolve the orders first
   (cancel, re-source, or transfer stock in), then count again.
5. **Review.** *Stok hareketleri* → filter **Sayım düzeltmesi** to see every
   correction a count has applied. `GET /inventory/stock-counts` gives the counts
   themselves, including the ones that found no difference.
6. **Reconcile.**

   ```bash
   python scripts/reconcile_inventory.py --store-id 2
   python scripts/reconcile_inventory.py --json          # machine-readable
   ```

   Exit code `0` means every summary matches its ledger and its order lines, every
   transfer has both legs, and every count has exactly the movement its delta
   demands. Exit code `1` names what is wrong — including the `stock_count_id` of any
   count whose movement is missing or does not match.

   It never writes. A reconciliation that "fixed" drift by overwriting the summary
   would destroy the evidence needed to find the bug that caused it.

## 16. Migration

Revision **`a3d7e9c14b62`** (`physical stock count workflow`), on top of
`f4b8c1d90e26` (inventory transfer workflow).

Purely **additive**. No existing row of any table is read, rewritten or deleted —
there are no counts to backfill, because before this migration a count was not a
thing that could be recorded.

A historical `MANUAL_ADJUSTMENT` that was *morally* a count is deliberately left
exactly as it is. Retro-labelling it a count would fabricate two system snapshots
that were never captured, which is precisely the lie this table exists to prevent.

`downgrade()` removes only this branch's schema — the table, the column, the
constraints, the indexes and the two triggers — and restores the previous movement
type domain and delta rule verbatim. It **refuses to run while any count exists**:
dropping the table would destroy the only record that a shelf was ever physically
counted, while leaving the stock those counts moved exactly where they moved it. The
ledger would be left with a bare −0.350 kg and nothing to explain it, and the
`STOCK_COUNT_ADJUSTMENT` rows could not be expressed in the restored type domain. A
lossy downgrade is worse than no downgrade. Export the counts first.

The append-only function `public.sweetops_block_inventory_mutation()` is **borrowed**
from `c3b7e01f9a24`, not owned by this migration, and is therefore not dropped on
downgrade — the movement ledger's own immutability trigger still depends on it.

Alembic remains **single-head**.

## 17. Deferred

Explicitly **not** in this branch, and not pretended at in the schema:

* **Approval workflow.** No `DRAFT`/`SUBMITTED`/`APPROVED` status. A status column
  with no state machine behind it is a lie. When approval is genuinely built, the
  status domain widens then.
* **Multi-person count.** One counter per count (`counted_by_user_id`). Blind
  double-counting, where two people count independently and the system compares them,
  is a real control — and a different feature.
* **Barcode / handheld scanning.** The count is typed.
* **Lot / expiry tracking.** A count is of a quantity, not of batches with dates.
* **Offline count mode.** The form needs the network; there is no local queue.
* **Scheduled cycle counts.** No recurring count plan, no "these 12 SKUs are due
  today". Counting is initiated by a human.
* **Supplier management / purchase orders.** Unchanged.
* **Full warehouse cycle-count system.** This is a branch waffle shop, not a
  distribution centre.
